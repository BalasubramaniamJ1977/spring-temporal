1 · Project Overview

This project is a reference implementation of a payment processing pipeline with production-grade observability baked in from day one. Eight Spring Boot microservices simulate a real-world cross-border payment flow — from channel initiation through routing, format transformation, fraud screening, accounting, posting, network dispatch, and final settlement response.

Every payment carries a UETR (Unique End-to-End Transaction Reference, a UUID defined by ISO 20022 / SWIFT gpi). The UETR is propagated through all eight services as both an OpenTelemetry span attribute (payment.uetr) and as an MDC key in structured logs, so a single identifier can retrieve the complete trace, all service logs, and span metrics in Grafana.

Goals

🔍 End-to-End Traceability

Paste one UETR into Grafana and instantly see the full 8-hop flame graph, every log line from every service, and per-stage latency — no manual correlation needed.

📊 Span-Derived Metrics

Tempo's metrics generator converts trace spans into Prometheus-compatible span metrics and service-graph metrics — no Prometheus client code in services.

🚀 Zero-Code Instrumentation

The OTEL Java agent auto-instruments Spring MVC, RestTemplate, and Logback. Services contain no OpenTelemetry imports except the shared UETRTraceFilter.

🐳 One-Command Stack

docker compose up starts all 14 containers — infra, services, and load generator — with all wiring pre-configured.

2 · Payment Pipeline

The pipeline is fully synchronous and linear. Each service receives an HTTP POST, enriches the payment JSON, and immediately calls its downstream neighbour before returning. The response therefore travels back through the same chain in reverse, so a single client call to payment-initiation blocks until all eight stages complete.

Payment Payload Lifecycle

StageServiceFields Added to PayloadNotes
0 Generate load-generator uetr, amount, currency, debtor_account, creditor_account, channel UETR seeded as OTEL trace ID
1 Initiate payment-initiation — Entry point; logs and forwards to routing
2 Route routing routing_rail (SWIFT / SEPA / CHAPS / BACS / FPS / RTP) Random rail selection
3 Transform transformer message_format (MT103 / pacs.008 / ISO20022 / …) Maps rail → ISO message format
4 Process payment-processor fraud_score, compliance_status Gaussian fraud model; score ≥ 0.7 → 422 REVIEW error
5 Account accounting-generator journal_id, debit_entry, credit_entry Generates double-entry ledger IDs
6 Post posting posting_ref Derives posting reference from journal_id
7 Dispatch dispatcher network_endpoint, dispatch_ref Maps rail to simulated network endpoint
8 Respond response-handler status: completed, new_uetr, full summary Terminal node; assembles final settlement response
Fraud path: When payment-processor assigns fraud_score ≥ 0.7 (roughly 5–10% of payments based on the Gaussian distribution), it throws HTTP 422. The error propagates back through the chain and is surfaced in Grafana as an error span, incrementing the Total Errors KPI and coloring the service map edge red.

Build — Multi-Stage Dockerfile

A single services/Dockerfile builds any of the eight services using a SERVICE_MODULE build argument:

# Stage 1 — Maven build (eclipse-temurin:21)
FROM maven:3.9-eclipse-temurin-21 AS build
ARG SERVICE_MODULE=payment-initiation
COPY pom.xml .
COPY common/pom.xml common/pom.xml        # always needed (shared library)
COPY ${SERVICE_MODULE}/pom.xml ...        # target service POM
RUN  mvn dependency:go-offline            # dependency layer cached separately
COPY common/src  common/src
COPY ${SERVICE_MODULE}/src ${SERVICE_MODULE}/src
RUN  mvn package -pl ${SERVICE_MODULE} -am -DskipTests

# Stage 2 — Runtime (eclipse-temurin:21-jre)
FROM eclipse-temurin:21-jre
RUN  curl -o /otel/javaagent.jar \
     opentelemetry-javaagent-v2.3.0.jar  # OTEL agent downloaded at build time
COPY --from=build .../${SERVICE_MODULE}/target/*.jar app.jar
ENTRYPOINT ["java", "-javaagent:/otel/javaagent.jar", "-jar", "app.jar"]
The Maven dependency cache is its own Docker layer — only invalidated when a POM file changes. Source-only changes rebuild in seconds.

4 · Common Library (services/common)

The common Maven module is a Spring Boot auto-configuration library. It is compiled into every service via -am (also-make) in the Maven build. No service needs to import or configure anything — Spring Boot's autoconfiguration mechanism picks it up automatically via META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports.

UETRTraceFilter

Registered at Ordered.HIGHEST_PRECEDENCE (outermost filter in the chain). On every inbound POST application/json it:

  1. Buffers the request body (so it can be read twice — once by the filter, once by Spring MVC).
  2. Extracts the uetr field from the JSON payload.
  3. Sets MDC["payment.uetr"] = uetr — appears in every Logback log line for the request.
  4. Sets span.setAttribute("payment.uetr", uetr) on the current OTEL span — enables the Grafana TraceQL query { span.payment.uetr =~ ".+" }.

CommonAutoConfiguration — Beans Provided

BeanTypePurpose
uetrTraceFilter FilterRegistrationBean<UETRTraceFilter> UETR → span attribute + MDC, all routes
restTemplate RestTemplate Built via RestTemplateBuilder; the OTEL Java agent instruments it to inject traceparent / tracestate headers automatically on every outbound call — no manual propagation code needed.
DownstreamErrorForwarder @RestControllerAdvice Catches HttpStatusCodeException and re-throws with the original HTTP status code so a 422 from payment-processor reaches the client as 422, not 500.

5 · Load Generator

A Python 3 async service (load_generator/generate_payments.py) that sends synthetic payments at a configurable rate (default: 10/min). Its key design point is that the UETR UUID is used as the OTEL trace ID, so the Grafana search by UETR finds the exact trace in Tempo without any additional look-up.

UETR → Trace ID Seeding

# UETR is a UUID, e.g. b4f48063-46e8-4eba-9f56-5d56e791fad3
uetr_hex = uetr.replace("-", "")           # 32-char hex = 128-bit integer
_uetr_trace_id.set(int(uetr_hex, 16))      # stored in a ContextVar

class UETRIdGenerator(IdGenerator):
    def generate_trace_id(self) -> int:
        tid = _uetr_trace_id.get()         # consume on first call
        if tid is not None:
            _uetr_trace_id.set(None)
            return tid                     # root span gets trace_id == uetr_hex
        return random.getrandbits(128)     # child spans: random

W3C Trace Context Propagation

After creating the root span, the generator injects the W3C traceparent header into the HTTP request to payment-initiation. The OTEL Java agent in each downstream service reads this header and continues the same trace, so all 8 services appear as child spans in a single flame graph rooted at the generator's span.

Tempo stores every span from all eight services. Its metrics generator derives two additional metric sets from the trace data and remote-writes them to Prometheus, enabling Grafana's service map panel and span-level latency/error metrics without any Prometheus client code in services.

Trace Drill-Down Flow

How to trace a specific payment:
1. The Payment Trace IDs table (left panel) lists the 20 most recent UETR-tagged traces.
2. Clicking any traceID cell sets the selectedTraceId URL variable.
3. The Call Trace panel (right, 12-column wide) re-queries Tempo with queryType: traceql + the hex trace ID, which Tempo's datasource detects as a trace ID lookup (isTraceIdQuery()) and fetches the full span tree.
4. The flame graph renders inline within the dashboard — no navigation to Explore needed.

Observability Signal Summary

🔥

Traces

Full 8-hop flame graphs per payment, stored in Tempo. Searchable by UETR via TraceQL span.payment.uetr attribute.

📈

Metrics

Span metrics and service-graph metrics auto-generated by Tempo. Scraped by Prometheus. No Prometheus client code in any service.

📋

Logs

Structured JSON logs from all 8 services, shipped via OTLP to Loki. Each log line carries payment.uetr in MDC for correlation.
